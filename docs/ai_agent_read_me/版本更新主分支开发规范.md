# 版本更新主分支开发规范

本文用于约束热更新模块依赖的主分支开发、版本号提交、PR 合并和 tag 发布流程。后续 AI 处理版本发布、打包或热更新相关任务时，应先阅读本文。

## 核心原则

- 普通用户更新目标必须是 `main` 上带版本 tag 的 PR merge commit。
- 开发者更新列表可以展示 `main` 上未打 tag 的 PR merge commit。
- 不把本地个人分支 merge commit 当作发行版本点。
- 不在用户运行目录原地执行 `git pull`、`reset --hard`、`clean` 或切换分支。
- 发行 tag 必须打在 GitHub PR 合并后生成的 merge commit 上，而不是打在个人分支、本地合并提交或普通功能提交上。

## 为什么必须 tag PR merge commit

热更新模块通过 GitHub API 读取 tag，并要求 tag 指向一个 PR merge commit。这个 commit 的提交信息通常形如：

```text
Merge pull request #939 from StareAbyss/xxx
```

这样软件才能稳定获得：

- 版本 tag，例如 `v3.1.1`
- PR 编号
- PR 标题
- PR 链接
- 对应 commit

如果在 `stareAbyss` 等个人分支上合并 `origin/main` 后打包，HEAD 往往是：

```text
Merge remote-tracking branch 'origin/main' into stareAbyss
```

这不是 GitHub PR merge commit，热更新状态文件无法从提交信息里解析出 PR 编号。

## 主分支合并规范

仓库应长期保持以下规则：

- `main` 的变更通过 PR 合并。
- 版本发布 PR 使用 `Create a merge commit`，不要用 squash merge 或 rebase merge。
- 禁止直接 push 到 `main`。
- 禁止 force push 到 `main`。
- 一个面向用户的功能尽量对应一个 PR，便于更新列表按功能展示。
- 版本号更新可以作为独立 PR，PR 中只修改全局版本号。
- 发行相关 PR 需要添加仓库现有标签 `Relese`。注意标签名是仓库中的实际拼写，不是 `release`。

如果合并方式改成 squash 或 rebase，`main` 上可能不再存在可识别的 PR merge commit，热更新模块的版本识别规则需要同步调整。

## 版本号更新标准流程

以发布 `v3.1.1` 为例：

1. 确认本地没有把无关改动混入版本号分支。
2. 从最新 `origin/main` 新建临时分支，例如 `codex/version-v3.1.1`。
3. 只修改 `function/globals/EXTRA.py` 中的 `VERSION`。
4. 提交和 PR 标题都使用“旧版本 -> 新版本”格式：

```text
build(版本号更新): v3.1.0 -> v3.1.1
```

5. 推送分支并创建 PR，base 为 `main`，并添加 `Relese` 标签。
6. 合并 PR。若仓库规则要求审核，管理员可以按项目规则绕过。
7. 拉取最新 `origin/main`。
8. 找到该 PR 在 `main` 上生成的 merge commit。
9. 在这个 merge commit 上创建 tag：

```powershell
git tag v3.1.1 <merge_commit_sha>
```

10. 单独推送 tag：

```powershell
git push origin v3.1.1
```

## 打包前检查

打包发行版前应确认：

- 当前要记录的版本号来自 `function/globals/EXTRA.py`。
- 版本 tag 已存在于远程。
- tag 指向 GitHub PR merge commit。
- 工作区没有会影响发行包判断的本地改动。
- `update_cache/update_state.json` 记录的 commit、tag、PR 与预期一致。

标准目录约定：

- 项目根目录的上一级可以放 `_biuld/`，作为临时构建工作区。
- `_biuld/` 内的临时 worktree 或源码拷贝可在打包结束后清理。
- 项目根目录的上一级放 `_dist/`，作为发行包输出目录。
- `_dist/` 是产物归档区，只能新增，不能由打包脚本删除旧产物。
- 发行包目录命名为 `FAA-v版本号 yyyy-MM-dd HH-mm-ss`。Windows 文件夹名不能包含英文冒号，所以时间使用 `HH-mm-ss`。

`一键生成分发资源.py` 已内置发行打包前置检查。不满足以下条件会直接中断，不会继续清理 `dist/` 或复制文件：

- 当前目录必须是 Git 工作区根目录。
- 当前 HEAD 必须有精确匹配的版本 tag。
- 当前 tag 必须与 `function/globals/EXTRA.py` 中的 `VERSION` 一致。
- 当前 HEAD 必须是 GitHub PR merge commit，提交信息应包含 `Merge pull request #数字`。
- 当前工作区必须干净，不能有本地修改或未跟踪文件。

脚本默认从自身所在项目根目录打包。若需要从 `_biuld/` 中的干净 tag worktree 打包，可设置：

```powershell
$env:FAA_PACKAGE_SOURCE_ROOT = "F:\...\_biuld\FAA-v3.1.1"
uv run python 一键生成分发资源.py
```

这种方式会从 `FAA_PACKAGE_SOURCE_ROOT` 读取源码并检查 Git 状态，但仍按源码所在位置推导输出到同级 `_dist/`。

如果必须在个人分支打包，要理解当前打包脚本记录的是当前 HEAD。个人分支 HEAD 不是 PR merge commit 时，PR 可能显示为未记录。

## 常用核查命令

查看版本号：

```powershell
Select-String -Path function\globals\EXTRA.py -Pattern "VERSION"
```

确认某个 commit 是否为 PR merge commit：

```powershell
git show --no-patch --pretty=%B <commit_sha>
```

确认 tag 指向：

```powershell
git rev-parse v3.1.1
git show --no-patch --pretty=%B v3.1.1
```

确认远程 tag：

```powershell
git ls-remote --tags origin v3.1.1
```

查看 PR merge commit：

```powershell
gh pr view <pr_number> --json mergeCommit,mergedAt,state,url
```

确认发行标签是否存在：

```powershell
gh label list --search Relese
```

## 注意事项

- 不要把 tag 打在 `stareAbyss` 本地 merge commit 上。
- 不要把 tag 打在版本号提交本身上，除非该提交已经是 GitHub PR merge commit。
- 不要在脏工作区直接创建版本号 PR，容易混入资源、配置或测试文件。
- 不要把发行 PR 标签写成 `release`；当前仓库使用的标签名是 `Relese`。
- 版本号 PR 合并后，应把最新 `origin/main` 合并回个人分支，再继续开发。
- 远程 tag 推送后，如发现 tag 打错，不要直接覆盖，应先和仓库维护者确认处理方式。
