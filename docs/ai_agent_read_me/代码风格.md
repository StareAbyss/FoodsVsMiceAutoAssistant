# 代码风格说明

本文件用于给后续 AI 智能体说明本项目的代码注释习惯。
若需要补充注释、docstring 或说明性块注释，请优先遵循本文件，而不是套用通用英文库的注释风格。

## 总体原则

- 注释默认使用中文，除非上下文已经是英文专有名词、库名、参数名或协议字段。
- 注释的目标是解释业务意图、流程分段、异常原因和数据结构，不要逐行翻译代码。
- 复杂流程可以多写注释；简单赋值、显而易见的循环、普通条件判断不要强行补注释。
- 采用Google 风格，允许AI对不符合 的注释直接修改风格

## 函数 docstring 风格

函数说明使用三引号 `""" ... """`，放在函数体第一行。
优先使用 Google 风格 docstring。常见格式如下：

```python
def some_function(arg_a, arg_b=None):
    """
    简短说明这个函数做什么，必要时补充业务场景或调用阶段。

    如果函数流程复杂，可以继续写第二行说明关键规则。

    Args:
        arg_a: 参数含义，必要时说明类型、取值范围、特殊值。
        arg_b: 参数含义，默认值代表什么。

    Returns:
        返回值含义；如果是状态码，需要写明每个状态码代表什么。
    """
```

具体约定：

- 第一段先写业务用途，不要只写“处理数据”“执行操作”这类空泛描述。
- 使用 `Args:` 描述参数，参数名必须和函数签名一致。
- 使用 `Returns:` 描述返回值。返回值是 tuple、dict、状态码、可为 `None` 时，要写清楚结构和异常含义。
- 如果返回状态码，按项目现有风格写在 `Returns:` 后，例如 `0-正常结束 1-重启本次 2-跳过本次`。
- 如果函数会主动抛出特定异常，使用 `Raises:` 描述异常触发条件。普通兜底日志不必强行写 `Raises:`。
- 私有辅助函数也应有简短 docstring，尤其是兜底、文件读写、线程、多进程、图像识别、DAG 排序等容易被误改的逻辑。
- 嵌套函数可以使用一行 docstring，但要说明它在外层流程中的判定目的。
- 旧代码中已有 Sphinx/reST 风格的 `:param:`、`:return:` 时，允许统一重写。

## 长流程注释

对于 `todo.py` 这类长流程函数，项目习惯使用三引号块注释分隔阶段，例如：

```python
"""检测是否成功进入房间"""
...
"""修改卡组"""
...
"""数据基础校验, 构建有向无环图, 完成高级校验"""
```

使用规则：

- 阶段注释用于标记流程大段落，适合战斗主流程、任务流程、识图结算流程。
- 阶段注释应描述“当前阶段要完成的业务目标”，不要写成实现细节。
- 如果阶段里包含多线程、截图识别、文件写入、上传服务器等副作用，需要在阶段开头或关键分支补充原因。
- 不要把阶段注释替代函数 docstring。函数入口仍应说明参数和返回。

## 行内和块内注释

项目中常见短注释风格：

```python
# 默认表示识别异常
result_drop_by_dict[p_id] = {"loots": None, "chests": None}

# 使用战利品计数list, 更新item_dag_graph文件, 不包含识别失败
best_match_items_success = [
    item for item in copy.deepcopy(list(loots_dict.keys())) if item != "识别失败"]
```

使用规则：

- 行上注释优先，用于解释下一小段代码。
- 行尾注释只用于非常短的补充，例如数据结构形状、特殊状态、单位。
- 注释应说明“为什么这样做”或“这个变量代表什么业务状态”，少写“遍历列表”“调用函数”。
- 对数据结构建议直接给例子，尤其是嵌套 dict/list，例如：

```python
# result_drop_by_list = {
#   1: {"loots": ["物品", ...], "chests": ["物品", ...]},
#   2: {"loots": ["物品", ...], "chests": ["物品", ...]},
# }
```

## 异常和兜底逻辑注释

涉及文件读写、JSON、线程、多进程、截图识别、服务端上传时，注释要写清楚兜底原因。

推荐写法：

```python
"""
读取主排序文件失败时, 使用 resource/template 中的备份文件兜底.

该逻辑用于避免 item_ranking_dag_graph.json 被清空或损坏时导致战斗结算中断.

Args:
    json_path: config 中的可写排序文件路径。

Returns:
    排序数据, 备份也不可用时返回空结构。
"""
```

注意：

- 不要只写“防止报错”，要写防止哪类报错、为什么可以继续运行。
- 如果兜底会丢失数据、跳过上传、放弃本次记录，应明确说明。
- 文件写入若使用临时文件、锁、原子替换，应说明是为了避免并发或中断导致主文件损坏。

## 日志文本风格

项目日志常使用带模块名的中文前缀，例如：


```python
CUS_LOGGER.warning(f"[战利品识别] [有向无环图] item_ranking_dag_graph.json 更新失败!")
```
- 详情查阅 CUS_LOGGER 全局变量
- 保持 `[模块] [子模块]` 风格，方便搜索日志。
- 日志中包含关键路径、玩家编号、关卡 ID、失败原因等定位信息。

## 给 AI 的再次提醒

- 不要把项目中已有中文注释改成英文。
- 不要删除带有业务吐槽但实际解释历史原因的注释，除非用户明确要求清理。
- 新增函数时，至少写清楚用途、参数、返回；新增复杂分支时，写清楚为什么存在这个分支。
- 修改 `analyzer_of_loot_logs.py`、`todo.py`、战斗流程、任务流程、图像识别流程时，注释要偏业务解释，方便后续 AI 快速理解调用链和失败兜底。
